1. Executive Summary

AEGIS (Agentic Exploit & Guardrail Investigation Suite) is an automated security testing platform that evaluates AI agents for vulnerabilities. It sends adversarial attack payloads at an AI agent equipped with tools (file access, database queries, email, code execution) and measures whether the agent can be tricked into performing malicious actions.

This assessment tested a qwen3:4b AI agent with 68 core attack payloads (86 total including extended set) across 7 attack categories, aligned with OWASP and MITRE security frameworks. Five defense modules were evaluated individually and in layered combinations.

Key Finding: Without defenses, 42.65% of all attack payloads succeeded (29 out of 68 core payloads). With the recommended layered defense configuration, the success rate dropped to 8.82%—a 79.31% improvement. Six sophisticated attacks remain unmitigated and require architectural-level countermeasures.

Top 3 Recommendations

  1. Deploy layered defenses immediately — Enable input_validator + output_filter + tool_boundary as the mandatory baseline for all production AI agent deployments.
  2. Restrict tool permissions — Tighten the permission enforcer to deny dangerous cross-tool flows (e.g., database→email, filesystem→HTTP) and enforce least-privilege access.
  3. Implement memory integrity checks — The remaining 6 unmitigated attacks exploit memory/context poisoning; add provenance verification for conversation history and retrieved documents.

Finding Details

F-1: Command Injection via MCP Tools Critical

What this means: An attacker could inject operating system commands through the AI agent’s tool interface. For example, by crafting a request that includes hidden shell commands, an attacker could execute arbitrary code on the server, read sensitive files, or take control of the system.

Evidence: 10 out of 10 command injection payloads succeeded without defenses. Techniques included semicolon injection, pipe chaining, command substitution, path traversal, and newline injection.

Business Impact: Complete system compromise, data exfiltration, service disruption.

F-2: Tool Misuse & Exploitation Critical

What this means: An attacker could trick the AI agent into misusing its legitimate tools in harmful ways—reading files it shouldn’t access, sending unauthorized emails, performing destructive database operations, or escalating beyond its intended scope.

Evidence: 9 out of 10 tool misuse payloads succeeded. Attacks included parameter injection, destructive operations, scope escape, and email abuse.

Business Impact: Unauthorized data access, data loss, reputational damage from unauthorized communications.

F-3: Supply Chain Vulnerabilities Critical

What this means: An attacker could compromise the AI agent by poisoning its tool definitions, injecting malicious instructions through retrieved documents, or substituting trusted data sources with compromised ones.

Evidence: 9 out of 10 supply chain payloads succeeded. Attacks included evil tool injection, RAG poisoning, rug-pull tool substitution, and cross-tool chain exploitation.

Business Impact: Persistent backdoor access, trusted channel compromise, cascading failures across connected systems.

F-4: Agent Goal Hijacking Medium

What this means: An attacker could attempt to redirect the AI agent’s objectives, overriding its intended purpose. The agent demonstrated resilience against most hijacking attempts but was susceptible to one targeted technique.

Evidence: 1 out of 10 goal hijacking payloads succeeded (direct override). Persona manipulation, gradual drift, and priority reordering attacks were blocked.

Business Impact: Agent performing unintended actions, potential misuse of resources.

F-5: Prompt Injection, Code Execution, Memory Poisoning Low

What this means: The agent demonstrated strong baseline resistance to direct prompt injection (0/13), unexpected code execution (0/10), and memory poisoning (0/5). These categories are currently well-controlled by the agent’s default safety features.

Note: Baseline ASR of 0% does not guarantee immunity. These categories should continue to be monitored as attack techniques evolve.

Recommended Defense Configuration

Deploy as mandatory baseline: input_validator + output_filter + tool_boundary
This three-layer configuration provides defense-in-depth across attack stages:
  1. Input Validator — blocks injection and hijacking attempts at the prompt boundary
  2. Tool Boundary — catches parameter-level abuse that bypasses input validation
  3. Output Filter — provides a final check on response content (defense-in-depth)

Residual Risk

Six attacks (8.82%) remain successful even with all five defenses active. These are sophisticated, multi-stage attacks that bypass all input and parameter boundaries:

  • Progressive drift attacks (ASI06) — gradually shift agent behavior over multiple conversation turns, evading per-turn validation
  • Supply chain poisoning (ASI04) — inject malicious instructions through tool outputs and retrieved documents, not user input
  • Context fabrication (ASI06) — create fake conversation history to manipulate agent behavior

These residual attacks require architectural mitigations: memory integrity verification, document provenance validation, and cross-turn behavior analysis.

8. Appendix

A. Methodology Summary

AEGIS uses automated black-box testing against a target AI agent with configured MCP tool surfaces. The assessment follows a structured approach:

  1. Baseline measurement — Run all attack modules with no defenses active to establish the unprotected attack surface.
  2. Single-defense evaluation — Activate one defense at a time and measure its individual impact on attack success rates.
  3. Layered-defense evaluation — Test combinations of defenses to measure defense-in-depth effectiveness.
  4. Delta comparison — Quantify each configuration’s improvement: delta = defended_ASR − baseline_ASR.

Scoring: Rule-based deterministic scoring (primary) with optional LLM-judge adjudication. Attack outcomes are binary: success (exploit objective achieved) or failure (probe blocked). Consensus scoring resolves disagreements using a configurable winner strategy.

D. Limitations

  • This is a point-in-time assessment; new attack techniques may emerge after the execution date.
  • Coverage is bounded by configured attack modules and payload sets. Real-world attacks may use techniques not covered.
  • Heuristic scoring can produce false positives and false negatives.
  • Results reflect the specific target model (qwen3:4b); different models may produce different vulnerability profiles.
  • Model non-determinism means exact metrics may vary between runs; trends and relative comparisons are reliable.
